# ADR-003: Use SQLite as the Persistent Data Store

**Status:** Accepted
**Date:** 2026-04-20
**Relates to:** ADR-001, ADR-002

## Context
The project brief in Section 3.3 requires persistent data storage and lists SQL Server, MongoDB, or "another persistent data store" as acceptable options. KnockKnock.prof needs to persist students, lecturers, consultation bookings, lecturer availability configurations, and an activity log that records every user action with a timestamp.

Our data is highly relational: consultations belong to lecturers, students join consultations, availability rules constrain valid bookings, and the activity log references users. The scale of the system is small — a 4-sprint group project with one deployed instance on Render (see ADR-002) and a user base limited to demonstration and marking.

Several categories of database were considered:

- **Client-server relational** (SQL Server, PostgreSQL, MySQL) — the DBMS runs as a separate process that the application communicates with over the network.
- **Document-oriented** (MongoDB) — stores data as flexible, nested documents rather than related tables.
- **Embedded relational** (SQLite) — the database engine is a library linked into the application, and the database itself is a single file.

## Decision
KnockKnock.prof will use **SQLite** as its persistent data store, accessed from Node.js via the `better-sqlite3` library.

## Rationale

**Why relational over document-oriented.** Our domain is naturally relational. A consultation references a lecturer, has many attending students, and must be validated against that lecturer's availability rules. These are join-heavy queries that relational databases handle natively through foreign keys and SQL. MongoDB's strength — flexible, nested, schema-less documents — does not align with our data shape, and would require us to either denormalise (risking inconsistency) or manually implement joins in application code.

**Why embedded over client-server.** Client-server databases like SQL Server and PostgreSQL are designed for very large datasets, high concurrency, and multi-user network access. Our deployment is a single Render instance serving a small number of users during demonstration and marking. A client-server DBMS would introduce operational complexity that the project does not need: a separate process to host, credentials to manage, a network boundary to secure, and another service that could fail independently of the application. SQLite eliminates all of this — the database is a single file that lives alongside the application.

**Why SQLite specifically.** SQLite meets all the goals of a DBMS described in the course material — efficiency, persistence, logging, and full ACID compliance (atomicity, consistency, isolation, durability). It uses standard SQL, so the team's query knowledge transfers to any relational database in future. The `better-sqlite3` Node binding provides synchronous, fast access with no configuration. Schema migrations are trivial and can be committed to the repository as plain SQL files.

## Consequences

**Positive:**
- Zero setup cost. The database is created automatically on first run; no separate installation, service, or credentials are required.
- Deployment is simple. The SQLite file is stored on Render's persistent disk, and no additional hosted service is needed.
- ACID guarantees are preserved. Concurrent booking attempts (for example, two students joining the last available slot) are safely serialised by SQLite's locking model.
- Standard SQL is used throughout, which aligns with the relational model taught in the course and keeps the team's knowledge portable.
- Testing is straightforward. Each Jest test can spin up an in-memory SQLite instance, giving fast, isolated tests with no shared state.

**Negative / tradeoffs:**
- SQLite is optimised for low-to-moderate concurrency. If KnockKnock.prof were scaled to multiple server instances behind a load balancer, SQLite's single-file model would become a bottleneck. At the project's intended scale this is not a concern, but it is the honest limitation of this choice.
- The team does not get hands-on experience with a client-server database or a document database in this project. This is an acceptable tradeoff given the project's focus on agile process, collaborative development, and core web technologies rather than database administration.
- Render's free-tier persistent disk must be configured correctly. A misconfiguration would cause the SQLite file to be wiped on redeploy. This risk is mitigated by explicit disk configuration in the Render service settings and by keeping a seed-data script in the repository.
